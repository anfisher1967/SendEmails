Critical Findings

Finding #1: Overly Permissive Owner Role

Denise Charron has been assigned the Owner role on mdc-d4ai-aoai-1, which grants "*" (wildcard) permissions. This violates the Principle of Least Privilege and allows unrestricted access including the ability to delete, modify, and share the resource without additional controls.

Finding #2: Excessive User Access

10+ users have Azure AI User roles providing direct access to the resource. Without clear business justification or role differentiation, this creates multiple attack vectors and increases the likelihood of credential compromise.

Finding #3: Credential Exposure on Devices

Four devices (demopc, alpineski-u4090, desktop-u475, ashhe003) contain stored Entra ID user cookies/tokens. If compromised, attackers could impersonate these users and gain lateral movement access to the Azure OpenAI resource.

Finding #4: Active Exploitation Attempts

We are observing ongoing jailbreak attempts from IP 52.177.6.190 (US, Virginia) targeting our gpt-4o deployment. These are Type: User Prompt Injection Attacks (UPIA) attempting to bypass safety guardrails.

Three Primary Attack Vectors

🔴 Vector 1: Direct User Access

10+ users have RBAC permissions for direct access to the resource. This includes Denise Charron with Owner role (most dangerous), and 9+ others with Azure AI User permissions.

🟠 Vector 2: Lateral Movement via Compromised Devices

Attack sequence: (1) Attacker compromises device, (2) Steals Entra ID user cookies/tokens, (3) Uses stolen credentials to authenticate as legitimate user, (4) Leverages user's permissions to access resource. Affected devices: demopc (High risk), alpineski-u4090 (Medium), desktop-u475 (Medium), ashhe003 (Low).

🔴 Vector 3: Active Exploitation (Jailbreak/Prompt Injection)

Ongoing attacks from 52.177.6.190 attempting User Prompt Injection Attacks (UPIA) to bypass AI safety guardrails. Objective is to extract sensitive data or manipulate model behavior.

💡 Strategic Recommendations

1. Implement Principle of Least Privilege (POLP)

📋 Action Items:

  • Remove Denise Charron's Owner role immediately and replace with Azure AI Developer role
  • Create custom role for resource-specific permissions if needed (limited to: read, write, deploy models only)
  • Audit all 10+ users and document business justification for each access grant
  • Right-size permissions - users should only have what they need for their role
  • Remove unnecessary Owner assignments at the resource level (keep at subscription/resource group level if operationally needed)

2. Role-Based Access Control (RBAC) Segmentation

📋 Proposed Role Structure:

  • Azure AI Developer: Deploy models, create deployments, read metrics (for development team)
  • Azure AI User: Query/use existing models, read results only (for end users)
  • Azure AI Cognitive Services User: Monitor and view resource health (for operators)
  • Custom Admin Role: Infrastructure changes only, require PIM approval (for platform team)
  • Owner: Reserved ONLY for subscription/resource group level, removed from individual resources

3. Just-In-Time (JIT) Privilege Escalation

📋 Implementation via Azure AD Privileged Identity Management (PIM):

  • Make all privileged roles "eligible" rather than "active" - users must request access
  • Require MFA + business justification for any escalation request
  • Set auto-expiration timers (e.g., 4-8 hours maximum for elevated access)
  • Enable approval workflows - manager or security team must approve escalations
  • Full audit trail - track who requested what, when, and why

4. Enforce Multi-Factor Authentication (MFA) & Conditional Access

📋 Policy Requirements:

  • Mandatory MFA for all users accessing mdc-d4ai-aoai-1
  • Device compliance requirement - only managed/compliant devices allowed
  • Location restriction - allow only from corporate network or approved VPN
  • Block legacy authentication - prevent weaker authentication methods
  • Block untrusted locations from accessing the resource
  • Require password change if sign-in risk is detected

5. Credential & Device Security

📋 Action Items:

  • Incident response for compromised devices - demopc (High), alpineski-u4090 (Medium), desktop-u475 (Medium)
  • Clear stored authentication tokens/cookies from all four devices
  • Force password reset for users who cached credentials on compromised devices
  • Enable Windows Hello / passwordless sign-in on corporate devices
  • Deploy Azure Key Vault for secret management instead of caching credentials

6. Monitor, Audit & Governance

📋 Implementation:

  • Enable Azure Policy to audit and flag assignments outside of approved roles
  • Set up alerts for any new Owner role assignments at the resource level
  • Quarterly access reviews - managers review and certify all user permissions
  • Monitor Azure activity logs for suspicious access patterns
  • Integrate with Security Information & Event Management (SIEM) for centralized monitoring
  • Establish SLA for remediation - non-compliant roles removed within 30 days

7. Threat Response (Active Jailbreak Attempts)

📋 Action Items:

  • Incident investigation - analyze patterns from IP 52.177.6.190
  • Enhance prompt filtering - strengthen Prompt Shields configuration
  • Rate limiting - implement stricter API rate limits to slow down brute-force attempts
  • IP blocking - consider blocking or throttling the source IP
  • Enhanced logging - capture all prompt injection attempts for forensic analysis
  • Model rollback - have a plan to quickly revert model if compromised
